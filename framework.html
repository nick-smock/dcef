DCEF / Framework

Systems Engineering
Framework

A standardized, tiered methodology for federated counter-UAS test and evaluation. This section covers project scope, the three-tier accuracy structure, data schema, evaluation team structure, and risk posture.

Project Overview

Why DCEF Exists

Counter-UAS systems are being fielded, evaluated, and procured across the United States at a pace that has outrun the methodology used to assess them. Range facilities, mobile evaluation teams, and government programs each generate their own test data — but without a shared standard for what that data means or how it was collected, no two reports can be meaningfully compared.

DCEF addresses this gap through a single, federated methodology anchored in onboard drone telemetry. By treating the flight controller's own logs — PX4 ULog and MAVLink tlog files from Pixhawk-based aircraft — as calibrated ground truth, DCEF creates a reference dataset that is independent of any specific sensor system, range facility, or vendor.

That ground truth is then correlated against the outputs of the counter-UAS system under test: radar tracks, optical detections, and acoustic signatures. The result is a standardized, reproducible evaluation record that can be compared across sites, reused across time, and cited in acquisition decisions.

Scope & Boundaries

What DCEF Covers

In Scope

  • Detection and identification performance evaluation of C-UAS sensor systems against small UAS threats
  • Onboard telemetry capture and standardization from Pixhawk-based flight controllers (PX4 ULog, MAVLink tlog)
  • Correlation of telemetry ground truth against radar, optical, and acoustic sensor outputs
  • Tiered positional accuracy requirements (Tier 1 GNSS through Tier 3 RTK with fixed base station)
  • Standardized data schema, time-synchronization protocol, and analysis pipeline
  • Synthetic example datasets and open-source Python toolchain for replication
  • Federated evaluation records suitable for cross-site and cross-vendor comparison

Out of Scope (Primary)

  • Kinetic effector defeat confirmation — DCEF records position truth for defeat geometry but does not adjudicate lethal outcomes
  • RF spectrum characterization as a primary deliverable — RF effects are captured incidentally through telemetry anomalies
  • Classified or export-controlled system performance data — all DCEF outputs are designed for public release
  • Prescriptive threat flight profiles — DCEF defines data standards, not operational tactics
  • Real-time battle management or command-and-control integration
Note on Non-Kinetic Effects

While RF jamming and GPS spoofing characterization are not primary DCEF deliverables, the telemetry record captures them as emergent data. When a jammer is activated, ULog and MAVLink logs surface packet loss rates, GCS disconnection events, and position drift in real time — without additional instrumentation. This data can be reported as a secondary output of any DCEF evaluation run.

Go/no-go detection evaluation at scale. Suitable for initial capability assessment across large vendor pools where absolute positional fidelity is secondary to detection rate.

No additional hardware investment beyond standard flight operations equipment. Enables broad participation across test sites without RTK infrastructure overhead.

TIER 02
≤30cm
CEP
Classification
Entry-Level RTK

Real-time kinematic positioning using a rover-only RTK receiver without a fixed base station. Network RTK or CORS corrections accepted.

RTK Rover · CORS / NTRIP corrections
Primary Use Case
Comparative Analysis

Meaningful side-by-side performance comparison between sensor systems and vendors. Sufficient positional fidelity to distinguish sensor track bias from true detection error.

Cost Rationale
Mid-Tier Investment

Moderate hardware cost with no fixed base station installation. Significant accuracy improvement over Tier 1 enables vendor differentiation that justifies additional outlay.

TIER 03
≤5cm
CEP
Classification
Full RTK + Fixed Base

Dual-frequency RTK with a surveyed fixed base station on-site. u-blox F9P recommended for both rover and base. Post-processed kinematic (PPK) supported.

u-blox F9P · Fixed base station
Primary Use Case
Kinetic Defeat Evaluation

High-fidelity position truth for kinetic effector assessment. At ≤5cm CEP, the ground truth record is precise enough to adjudicate near-miss geometry and defeat envelopes.

Cost Rationale
Ammunition Cost Reduction

Tighter positional CEP directly reduces rounds expended per conclusive evaluation run. For kinetic effector programs, the hardware investment is recovered through reduced ammunition expenditure across a test series.

Data Standard

Common Data Schema

Every DCEF evaluation record — regardless of tier, site, or sensor type — conforms to the following field set. Fields are sourced from either onboard telemetry, the sensor system under test, or derived through the correlation pipeline. This schema is the foundation of cross-site comparability.

Field Type Source Description
timestamp_utc datetime64 Telemetry UTC timestamp synchronized to GPS time from flight controller. Primary time axis for all correlation operations.
lat_truth float64 Telemetry Ground truth latitude from onboard GNSS / RTK receiver, in decimal degrees WGS84.
lon_truth float64 Telemetry Ground truth longitude from onboard GNSS / RTK receiver, in decimal degrees WGS84.
alt_agl_m float32 Telemetry Altitude above ground level in meters, derived from barometric and GNSS fusion in the flight controller EKF.
hdg_deg float32 Telemetry Vehicle heading in degrees true. Used for polar bias plot orientation.
gnd_speed_ms float32 Telemetry Ground speed in meters per second. Used for range-rate binning in detection performance analysis.
sensor_lat float64 Sensor Reported target latitude from the C-UAS sensor under test, in decimal degrees WGS84.
sensor_lon float64 Sensor Reported target longitude from the C-UAS sensor under test, in decimal degrees WGS84.
sensor_type string Sensor Modality identifier: RADAR, OPTICAL, ACOUSTIC, or RF. Enables multi-modal fusion analysis.
detection_flag bool Sensor Boolean detection event flag from the sensor system. True = sensor declared detection at this timestamp.
range_m float32 Derived Slant range in meters between sensor origin and ground truth position. Computed from truth lat/lon and sensor position.
pos_error_m float32 Derived Horizontal position error in meters: geodesic distance between sensor_lat/lon and truth lat/lon at each timestamp.
tier int8 Derived DCEF accuracy tier (1, 2, or 3) applied during this evaluation run. Governs CEP threshold for pass/fail assessment.
rf_packet_loss_pct float32 Telemetry Percentage of GCS telemetry packets lost during the evaluation interval. Non-zero values indicate RF interference or jamming events.
site_id string Derived Unique site identifier for federated record linkage. Enables cross-site dataset aggregation without merging raw files.

Risk Posture

Top Program Risks

The following risks represent the primary threats to DCEF evaluation fidelity and framework adoption. Each has been assessed for likelihood and consequence severity, with mitigations built into the framework design.

Risk Likelihood Consequence Mitigation
GNSS multipath or signal degradation at test site invalidates Tier 1/2 positional accuracy High High Pre-survey site HDOP conditions; mandate HDOP < 1.5 threshold in flight data before run acceptance. Tier 3 RTK base station mitigates at highest fidelity level.
Time synchronization drift between sensor output and telemetry log exceeds correlation tolerance Medium High Enforce GPS-disciplined NTP on all sensor recording systems. Define maximum allowable skew (±50ms) in schema specification; flag and exclude out-of-tolerance records.
Inconsistent sensor configuration across sites reduces cross-site data comparability High Medium DCEF schema mandates sensor_type and configuration metadata fields per record. Standardized pre-run checklist distributed to all participating sites.
Proprietary sensor output formats incompatible with DCEF CSV schema pipeline Medium Medium Open-source Python adapter library provided in DCEF Tools section. Vendors required to map outputs to schema fields; unmappable fields excluded rather than force-fitted.
Weather or airspace restrictions limit test execution windows and reduce statistical sample size Medium Low DCEF defines minimum run counts per tier for statistically significant detection probability curves. Schedule buffer and alternate site provisions built into test planning template.
