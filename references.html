[01] Government 2025
Close Combat Lethality Task Force (CCLTF) Annual Report — Fiscal Year 2025
U.S. Army · Headquarters, Department of the Army (HQDA G-3/5/7)
Describes CCLTF priority areas for FY25 including mobile C-sUAS, counter-sensing, and joint working group coordination. Documents formalization of the CCLTF under HQDA and validation of the FY26–30 Program Objective Memorandum. Provides operational context for the fragmented evaluation landscape that DCEF addresses.
View Report (PDF) · benning.army.mil
How DCEF Cites This
Used to establish the operational context and urgency of the C-UAS evaluation problem. The CCLTF FY25 report documents that C-sUAS coordination is occurring across multiple services and working groups without standardized evaluation methodology — precisely the fragmentation DCEF is designed to resolve.
Framework: Problem Statement

Congressional Research Service (CRS) · Congress.gov

Also used to anchor the threat classification structure (Groups 1–3 UAS) referenced in the DCEF tier definitions.

Cited to establish the legislative mandate for standardized C-UAS evaluation and policy coordination. [04] Regulatory 2025
C-UAS Grant Program — Notice of Funding Opportunity
Federal Emergency Management Agency (FEMA) · One Big Beautiful Bill Act, Pub. L. No. 119-21
Establishes the $500 million C-UAS Grant Program to enhance state and local capabilities to detect, identify, track, and monitor unmanned aircraft systems. Funds support U.S. state, local, tribal, and territorial governments in combatting unlawful UAS use. Demonstrates the scale of public investment in C-UAS detection and the need for standardized performance evaluation of recipient systems.
View NOFO · fema.gov
How DCEF Cites This
Cited to establish the scale of distributed C-UAS procurement now occurring at the state and local level. $500M flowing to non-federal entities with no standardized evaluation requirement creates exactly the cross-site comparability gap that DCEF addresses. Also anchors the economic argument for the tiered framework — smaller jurisdictions need a low-cost entry point.
Framework: Economic Justification
[05] Technical Current
ULog File Format Specification
PX4 Autopilot Project · PX4 Developer Documentation
Authoritative specification for the ULog binary file format used by PX4 firmware for flight logging. Defines the self-describing format that includes message types, parameter values, and timestamped uORB topic data. Specifies the flag bits structure, message headers, and data sections used by the pyulog parser to extract position, attitude, speed, and RF telemetry fields.
View Specification · docs.px4.io
How DCEF Cites This
The ULog format is the primary source of ground truth in DCEF Tier 1, 2, and 3 evaluations. This specification defines which data fields are available, their data types, and how they are timestamped — forming the technical basis for the telemetry extraction step in the DCEF pipeline and the schema fields sourced from flight controller logs.
Tools: parse_telemetry.py · Schema: lat_truth, lon_truth, hdg_deg
[06] Technical Current
MAVLink Developer Guide — Protocol Specification v2
MAVLink Project · mavlink.io
Defines the MAVLink communication protocol used between PX4 flight controllers and ground control stations. Specifies the publish-subscribe and point-to-point message architecture, common message set (common.xml), time synchronization protocol, and telemetry status messages used for RF link quality monitoring. MAVLink tlog files are ground-station-recorded binary logs of all MAVLink traffic.
View Documentation · mavlink.io
How DCEF Cites This
MAVLink tlog files provide a secondary telemetry source in DCEF, particularly for RF link quality data (packet loss, RSSI) not always present in ULog. The MAVLink time synchronization protocol underpins the DCEF 50ms time-skew tolerance requirement. The schema field rf_packet_loss_pct is derived from MAVLink telemetry_status messages.
Schema: rf_packet_loss_pct · Framework: Time Sync Protocol
[07] Technical 2024
ZED-F9P Product Summary — u-blox F9 High Precision GNSS Module
u-blox AG · UBX-17005151
Product summary for the ZED-F9P multi-band RTK GNSS module. Documents concurrent reception of GPS, GLONASS, Galileo, and BeiDou signals; centimeter-level accuracy with fast RTK convergence; RTCM 3.3 correction support for local base stations and virtual reference stations; and advanced jamming and spoofing detection via Galileo OSNMA. The recommended hardware for DCEF Tier 3 evaluations.
View Product Summary (PDF) · u-blox.com
How DCEF Cites This
The F9P is the specific hardware recommendation for DCEF Tier 3 (≤5cm CEP). The product summary provides the technical basis for the CEP claim and the fixed base station configuration required at Tier 3. The spoofing and jamming detection capability is also relevant to the RF packet loss field documented in the DCEF schema.
Framework: Tier 3 Hardware Requirement

[10] Technical Current
MAVLink Common Message Set — GLOBAL_POSITION_INT & TELEMETRY_STATUS
MAVLink Project · mavlink.io/en/messages/common.html
Defines the common MAVLink message types used for position telemetry and link quality reporting. GLOBAL_POSITION_INT (message #33) provides GPS-fused lat/lon/alt at the native flight controller rate. RADIO_STATUS provides RSSI, noise floor, and packet error counts used to compute the rf_packet_loss_pct field in the DCEF schema. Both are present in any MAVLink tlog file.
View Message Set · mavlink.io
How DCEF Cites This
Cited to specify the exact MAVLink message types that supply position and RF quality data in the DCEF pipeline. This reference makes the data source traceable to a versioned, open specification — meaning any evaluator can verify exactly where each schema field originates and replicate the extraction process independently.
Schema: rf_packet_loss_pct · Tools: parse_telemetry.py
